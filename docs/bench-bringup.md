# Community-firmware bench bring-up

How to take the community-firmware image (`firmware/build/fw/b288-community.hex`) from
"builds and links" to "makes sound" on a real 288r. Work top to bottom with an ST-Link and a logic
analyzer / scope. **Keep `Compiled FW/B288-REV1.0.hex` as the restore image at all times** — SWD +
the BOOT0 ROM bootloader always recover the module, so nothing here can brick it.

> **Read this first.** The image is a *structurally complete* bring-up: clock → SDRAM → codec →
> SAI/DMA → the validated smooth-delay engine, 8 taps out the 8 DAC channels. But every value tagged
> `[BENCH]` in `firmware/src/bsp/board.h` is a best-effort guess. **Do not expect audio on the very
> first flash** — expect to confirm/patch the constants below, one peripheral at a time. If you want a
> guaranteed-working hardware test *now*, flash the already-validated interpolation patch
> (`B288-REV1.0-interp-EXPERIMENTAL.hex`, [docs/bench-runbook.md](bench-runbook.md)) instead; the full
> rewrite is the next stage.

## Build & flash

```bash
cd firmware && make firmware          # -> build/fw/b288-community.{elf,hex,bin}
openocd -f interface/stlink.cfg -f target/stm32f4x.cfg \
  -c "program build/fw/b288-community.hex verify reset exit"
```

Bring-up is easiest with OpenOCD attached (`telnet localhost 4444`) so you can `halt`, read registers,
and set breakpoints as you go.

## Bring-up order (one peripheral at a time)

Each step has a **pass check** you can make over SWD before moving on. All the constants live in one
file: `firmware/src/bsp/board.h`.

### 1. Clock — `bsp_clock_init()` (`clock.c`)
Confirmed from bench 1 (HSE 8 MHz → 168 MHz), so this should just work.
- **Check:** after `halt`, read `RCC->CFGR` SWS = PLL; MCO or a scope on a GPIO toggle confirms
  168 MHz. If the core is dead here, HSE isn't starting — check the crystal.

### 2. SDRAM — `bsp_sdram_init()` (`sdram.c`)
`[BENCH]` items: which **FMC bank** the SDRAM CS is wired to (code assumes bank 1 @ `0xC0000000`),
the **FMC pin list**, and **CAS latency** (try 3 then 2).
- **Symptom if wrong:** boot hangs in `dl_clear()` (it writes 8 MB to SDRAM). If the module is dead
  after flashing, suspect SDRAM.
- **Check:** breakpoint after `bsp_sdram_init()`, then over SWD write/read a few words at
  `0xC0000000`, `0xC0400000`, `0xC07FFFFC`. A correct controller reads back what you wrote across the
  whole 8 MB. If a short pattern works but far addresses alias, the row/column config is off.

### 3. Codec — `bsp_codec_init()` (`codec_cs42888.c`, I2C1)
Bus is confirmed I2C1; the **address** (AD0/AD1 straps → 0x48–0x4B) and **register values** are
`[BENCH]`, and the **RESET pin** is not yet driven.
- **Do this:** sniff the *stock* firmware's power-up I2C with the logic analyzer — it gives you the
  exact address and the exact register writes. Transcribe them into `cs_write()` and
  `CS42888_I2C_ADDR7`. Also drive the codec RESET line high before configuring (add the GPIO in
  `bsp_codec_init()`; find the pin from the schematic/board).
- **Check:** `bsp_codec_init()` returns 0 (I2C ACKs). A NAK returns non-zero but boot continues, so
  you can probe the bus with the module running.

### 4. Audio path — `bsp_audio_init()` / `bsp_audio_start()` (`audio_sai.c`)
The heaviest `[BENCH]` block: **SAI pin AF mapping**, the **MCLK/bit-clock chain** (`PLLSAI_*` in
`clock.c` + `SAI_MCKDIV`) to actually land on 96 kHz, **FS framing** (FSALL/FSPOL/FSOFF/FSDEF), and
the **DMA stream/channel** the silicon maps SAI1_A/_B to. Also confirm whether the codec MCLK is
generated by the F429 SAI or by a **board oscillator** — if the board clocks the codec, the F429 SAI
runs as a slave and this section changes.
- **Check clocks first:** scope SAI SCK, FS, MCLK. SCK should be `96 kHz × 256 = 24.576 MHz`, FS at
  96 kHz. Retune `PLLSAI_N/Q/DIVQ` + `SAI_MCKDIV` until they match.
- **Check DMA:** breakpoint in `DMA2_Stream1_IRQHandler` — it must fire at `96 kHz / 16 ≈ 6 kHz`. If
  it never fires, the DMA request mapping (stream/channel) is wrong.
- **Check data:** feed a known tone into an input; over SWD inspect `rxbuf` for non-zero samples in
  the expected slot (set `AUDIO_IN_SLOT`).

### 5. Tap → output-slot order (`main.c` bridge)
The bridge scatters engine tap `t` to TDM slot `t`. The real **slot→jack order** is `[BENCH]`.
- **Check:** solo one tap (in a test build, gain only tap 0) and confirm which physical output jack
  carries it. Fix the mapping in the `bsp_audio_isr()` bridge if scrambled.

### 6. Panel switches (`gpio_panel.c`)
Pins are confirmed (cal PB10, cycle PB11, resolution PD11/12); **polarity** is `[BENCH]`.
- **Check:** toggle each switch, read the GPIO IDR over SWD, confirm the sense in `bsp_sw_*()`.

### 7. Make the multiplier live (the smooth-modulation payoff)
Two sources are already coded, both **off by default** so the audio path validates first. Enable
whichever the bench confirms, in `board.h`:
- **Time-CV** (`USE_CV_MULT 1`): the multiplier is read per-sample from a codec ADC slot
  (`MULT_CV_SLOT`, [BENCH] which of slots 1–3) inside the audio ISR — this is the path you modulate
  with a CV, and the one that shows off the smooth (non-zippered) sweep. Set `CV_MULT_OFFSET`/`SCALE`
  from the measured 0 V / full-swing readings.
- **Multiplier pot** (`USE_SPI_MULT 1`): read from the SPI2 control-surface ADC (`bsp_pot_read`,
  `spi2_adc.c`) in the superloop. [BENCH] confirm the ADC part (MCP320x assumed), the pin orientation,
  and which channel (`MULT_POT_CH`) is the multiplier.
- **Check:** with one enabled, a swept CV / turned knob should glide the delay time smoothly. If it
  jumps around, the slot/channel or the ADC protocol is wrong — fix the one constant and rebuild.

## What still isn't wired (expected — next layers, not bugs)

- **The rest of the control surface** beyond the multiplier: the 9 sliders + remaining pots (SPI2 ADC)
  and the 74HC595 DIP / 74HC4051 trimmer scan (preset banks, phase/mute, tap times).
- **Momentary-switch transport** (write/recirc gestures) and the **settings/calibration** boot chord.
- **Pitch mode** (`pitch_tap` dual-head reader), **analog tone**, and the **int16/int32 SDRAM layer** —
  staged rewrite modules (see `firmware/DESIGN.md`).

## When audio passes end-to-end

File the measured constants back into `firmware/src/bsp/board.h` (and the codec register sequence into
`codec_cs42888.c`), commit, and note the results in an issue. That closes the `[BENCH]` set and turns
this bring-up into a trustworthy image.
